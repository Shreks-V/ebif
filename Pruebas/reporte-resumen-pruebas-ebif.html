Introducción

Este documento presenta el reporte de las pruebas automatizadas del sistema, orientadas a verificar el comportamiento acordado en las historias de usuario y criterios de aceptación referenciados en el código mediante identificadores SV-*.

Las pruebas se ejecutan con pytest desde el repositorio del proyecto, utilizando una configuración integrada del código del backend (FastAPI) con módulos de prueba ubicados en la carpeta Pruebas/. Gran parte de las validaciones se realizan contra aplicaciones mínimas levantadas en memoria, con repositorios simulados, lo que permite ejecutar los casos de forma repetible en entornos de integración continua o en máquinas de desarrollo sin depender de infraestructura externa.

El alcance incluye autenticación y sesión, beneficiarios, citas, pre-registro, recibos y cobros, acciones rápidas del tablero (donde aplique) y comprobaciones sobre código fuente del frontend cuando el criterio se valida mejor de forma estática.

1 Introducción

1.1 Antecedentes

Este documento ofrece el resumen de las actividades de prueba correspondientes a la iteración Iteración documentación y pruebas automatizadas del sistema EBIF — Sistema de gestión (backend FastAPI + frontend Angular).

El sistema bajo prueba (AUT), EBIF, da soporte a flujos operativos de beneficiarios, citas, pre-registro, recibos y cobros, vistas relacionadas con inventario y acceso autenticado. Las pruebas automatizadas están implementadas con pytest en el paquete Pruebas/, ejercitando rutas FastAPI mediante repositorios en memoria cuando corresponde, de modo que no se requiere una base Oracle en vivo para la mayoría de los casos. La versión del repositorio documentada aquí corresponde a la rama de desarrollo utilizada para la entrega del curso (abril de 2026).

2 Panorama general

Esta sección ofrece una visión de alto nivel de los hechos y actividades registrados durante la iteración Iteración documentación y pruebas automatizadas sobre EBIF — Sistema de gestión (backend FastAPI + frontend Angular).

También delimita el alcance de las pruebas (qué se probó y qué no) y describe el entorno (hardware, software y datos).

Alcance probado. Pruebas automatizadas a nivel de API para autenticación y sesión, beneficiarios, citas, pre-registro, recibos (filtros, métodos de pago, cancelación y validaciones), y comprobaciones estáticas sobre código Angular del tablero (acciones rápidas y rutas protegidas cuando aplica).

Alcance no probado o parcial. Flujos de extremo a extremo en navegador (por ejemplo foco con teclado, activación con Enter, contraste cromático en acciones rápidas) aparecen como omitidos o aplazados a herramientas futuras (Playwright/Cypress o auditorías de accesibilidad). El comportamiento respaldado solo por procedimientos almacenados en Oracle se aproxima en memoria para repetibilidad; la integración contra base en producción queda fuera del alcance de esta iteración.

Por ejemplo Las pruebas de la iteración para EBIF se llevaron a cabo en abril de 2026 como parte de las entregas del curso. El esfuerzo se centró en regresión automatizada de endpoints REST con pytest y análisis estático puntual del frontend. La ejecución se realizó en equipos de estudiantes o desarrolladores (entornos típicos Windows 10/11) con Python 3.x, dependencias de backend/requirements-dev.txt y el proyecto configurado con pythonpath = backend . en pytest.ini.

3 Desviaciones

Esta sección documenta desviaciones respecto a lo acordado previamente, en especial cuando puedan afectar la aceptación de los resultados, incluyendo referencias a la documentación que explique los motivos.

Por ejemplo La plantilla NIST supone un artefacto formal de plan de pruebas con objetivos firmados; en esta iteración académica los objetivos detallados están embebidos en códigos de historia (SV-*) y docstrings de los módulos de prueba, no en un plan aparte. 6 Evaluación

Evaluación global del proceso de pruebas, problemas y limitaciones.

Por ejemplo Los módulos de backend bajo prueba reciben verificación repetible a nivel de API sin Oracle, lo que acorta el ciclo de retroalimentación. La principal limitación es de fidelidad: el comportamiento en memoria puede diferir levemente del de procedimientos almacenados en producción en casos límite. El riesgo se considera bajo para los objetivos de aprendizaje del curso, pero medio si este mismo conjunto fuera el único criterio de salida a producción sin pruebas de integración y UI adicionales.
